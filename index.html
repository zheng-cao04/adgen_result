Key Findings

1 Qwen3-VL-8B-Instruct critic catches basic failures, but its wording can be imprecise.

In the lipstick run, the critic correctly flagged random text, weak lipstick visibility, and unclear product action. But it described the missing action as "lipstick application or transfer"; that wording was ambiguous for this product, because the actual promise is no transfer to the cup, not visible lipstick transfer.

Qwen3-VL critic output for this video

{
  "pass": false,
  "summary": "A woman holds a white mug and turns it to show the inside, with a lipstick and its case on the table nearby.",
  "failure_reasons": [
    "Visible text overlay 'XNOCB' appears in the frame, which is unrequested.",
    "The woman is not shown wearing lipstick matching the product reference.",
    "The main product action (lipstick application or transfer) is not visible."
  ],
  "note": "The first two issues are useful. The third issue is directionally useful but semantically ambiguous: for this product, the intended proof is no lipstick transfer on the cup, not a visible transfer mark."
}
2 LTX2.3 often introduces unexplained subtitles or text artifacts.

Multiple generated clips show random captions, letters, or fake overlay text even when the prompt focuses on clean product action, controlled composition, and stable product visibility.

3 Video prompts work best when they are concise, direct, and positive.

The video prompt should describe the exact video we want to generate: subject, product, setting, camera, motion, and proof moment. It should not carry a separate negative-prompt constraint list; those lists add noise and did not reliably prevent model artifacts. The current job records therefore keep prompts as positive visual instructions only.

Current End-to-End Flow

The current AdGen demo path is a direct full-video workflow. The planner always creates one ad_blueprint.json first. The first branch is whether the brief includes product.image_path: product-image briefs create a direct start frame before video generation; prompt-only briefs skip that image stage. The second branch happens after the critic: pass accepts the run, fail writes a retry job and regenerates.

1. Set run inputs Choose BRIEF, create RUN_DIR, and point the image and video clients at the HTTP services.
2. Plan plan_ad.py writes a direct ad_blueprint.json with one positive full-video plan.
3. Branch: product image If product.image_path exists, Qwen Image Edit creates key_images/direct/01_full_start.png.
4. Build jobs build_video_jobs.py emits one provider-neutral request in video_jobs.json.
5. Generate video generate_video.py calls the LTX2.3 HTTP server and writes video/01_full.mp4.
6. Branch: critic Qwen3-VL either passes the clip or triggers a GPT rewrite plus retry video generation.

Command template

export BRIEF="examples/electronics/brief.json"
export BRIEF_NAME="$(basename "$(dirname "${BRIEF}")")"
export RUN_DIR="runs/${BRIEF_NAME}_$(date '+%Y%m%d_%H%M%S')"

export VIDEO_ROUTE=direct
export IMG_API_BASE="http://127.0.0.1:58084"
export VIDEO_API_BASE="http://127.0.0.1:58083"
export VIDEO_BLUEPRINT="${RUN_DIR}/ad_blueprint.json"

python scripts/plan_ad.py "${BRIEF}" "${RUN_DIR}" --execution-route "${VIDEO_ROUTE}"

# Branch A: product.image_path exists, so product image -> direct start frame -> video.
python scripts/generate_direct_start_frame.py "${RUN_DIR}/ad_blueprint.json" \
  --backend http \
  --api-base "${IMG_API_BASE}" \
  --model-id models/Qwen-Image-Edit-2511 \
  --size "704*1280" \
  --dtype bfloat16 \
  --steps 12 \
  --true-cfg-scale 4.0 \
  --guidance-scale 1.0 \
  --timeout-seconds 1800

export VIDEO_BLUEPRINT="${RUN_DIR}/ad_blueprint_with_direct_start_frame.json"

# Branch B: no product.image_path, so skip image-edit and keep:
# export VIDEO_BLUEPRINT="${RUN_DIR}/ad_blueprint.json"

python scripts/build_video_jobs.py \
  "${VIDEO_BLUEPRINT}" \
  "${RUN_DIR}" \
  --size "704*1280" \
  --fps 24 \
  --execution-route "${VIDEO_ROUTE}"

python scripts/generate_video.py "${RUN_DIR}/video_jobs.json" \
  --backend http \
  --api-base "${VIDEO_API_BASE}" \
  --timeout-seconds 7200

python scripts/critic_video.py "${RUN_DIR}/video_jobs.json" \
  --model-path models/Qwen3-VL-8B-Instruct \
  --only-role full \
  --fps 1 \
  --max-pixels 230400 \
  --output "${RUN_DIR}/critic_report.json"

# Critic pass: accept/archive the run.
# Critic fail: rewrite and regenerate.
python scripts/retry_video_from_critic.py \
  "${RUN_DIR}/video_jobs.json" \
  "${RUN_DIR}/critic_report.json" \
  --only-role full

python scripts/generate_video.py "${RUN_DIR}/video_jobs_retry_01.json" \
  --backend http \
  --api-base "${VIDEO_API_BASE}" \
  --timeout-seconds 7200

What brief.json is

brief.json is the pipeline input contract: a compact product and ad request supplied by the user.

brief.json fields
product.name
The product or service being advertised.
product.category
Optional planner hint such as beauty, food, electronics, software tool, fashion, or furniture.
product.selling_points
Benefits, constraints, proof points, or required demo actions the ad should preserve.
product.image_path
Optional product reference image. In the product-image path, AdGen sends it to image-edit/ref2img to create the keyimage used by video generation.
language
Output language for planner text and generated ad prompts.
duration_seconds
Target total video length for the generated ad.
execution_route
direct asks the planner for one full-video prompt. The result browser separates direct runs by whether they use only text or first create a product-conditioned keyimage.

High-level flow

From that brief, AdGen builds inspectable JSON artifacts at each step. The current result view compares two direct-generation paths: pure prompt generation, where one full-video prompt is sent to the video model, and product-image generation, where the product image is first turned into a keyimage that conditions the final video.

1. brief.json The user-facing input contract: product details, constraints, references, and duration.
2. ad_blueprint.json The GPT planner creates strategy, asset memory, and a route-specific execution layer, without model negative prompts.
3. route selection Choose either prompt-only video or product image to generated keyimage to video.
4. video jobs Build one provider-neutral full-video request, with or without a start keyimage.
5. video generation The HTTP backend generates the final mp4 from the prompt-only or keyimage-conditioned request.

Stage 1: brief.json -> ad_blueprint.json

The planner stage converts a compact user brief into ad_blueprint.json. The output shape now follows execution_route. Direct route asks GPT for one unified full-video prompt. Keyframe route asks GPT for hook/ad shots and key-image prompts. This stage no longer stores negative_prompt; later image/video job builders carry forward positive visual instructions. This stage still does not generate media.

Stage 5: generated mp4 -> critic_report.json

The critic stage evaluates generated clips against the compact checklist stored in video_jobs.json. It uses a local Qwen3-VL video critic to produce a structured pass/fail report. The current schema intentionally removes numeric scores and long evidence fields; it keeps only pass, summary, and failure_reasons per clip.

Command

python scripts/critic_video.py "${RUN_DIR}/video_jobs.json" \
  --model-path models/Qwen3-VL-8B-Instruct \
  --only-role full \
  --fps 1 \
  --max-pixels 230400 \
  --output "${RUN_DIR}/critic_report.json"

Input expectation shape

{
  "metadata": {
    "role": "full",
    "critic_expectation": {
      "purpose": "Judge whether the raw video is a plausible product demo for Portable LED Lamp.",
      "hard_requirements": [
        "single continuous raw video shot with no obvious cut, montage, split screen, or scene change",
        "main product demo action: the lamp is placed, turned on, and the warm light is visible"
      ],
      "soft_context": [
        "Focus the judgment on the requested product action, proof visibility, and one-take continuity."
      ]
    }
  }
}

Output: critic_report.json

{
  "pass": false,
  "clips": [
    {
      "job_id": "portable_led_lamp_df_v1_full",

      "pass": false,

Pass Accept generated clips and end the run.
Fail Use failure_reasons, the original prompt, and blueprint/request context as input to GPT prompt rewriting.
Retry Regenerate the failed role from video_jobs_retry_01.json, then run the critic again.

A separate key-image critic still exists for multi-candidate key-image selection, but it is not part of the direct route. Video critic failures are handled by scripts/retry_video_from_critic.py.

Stage 6: critic fail -> GPT retry

Retry no longer has a rule-based prompt patcher. A failed clip is retried by calling GPT with the original video prompt, critic summary/failure reasons, and blueprint/request context. GPT returns both start_image_prompt and video_prompt. For failed I2V/direct requests, the script generates a fresh retry start frame by default, then writes a retry video job that points at it. Use --skip-retry-start-frame only when the existing start frame should stay fixed.

Command

python scripts/retry_video_from_critic.py \
  "${RUN_DIR}/video_jobs.json" \
  "${RUN_DIR}/critic_report.json" \
  --only-role full \
  --retry-index 1 \
  --output "${RUN_DIR}/video_jobs_retry_01.json" \
  --plan-output "${RUN_DIR}/video_retry_plan_01.json" \
  --retry-start-frame-offload-mode sequential
